### 3. PDF EXTRACTOR — the first background agent

**Status:** Architecture sketched tonight. Not started.

**Goal:** Daily / weekly automated download of high-impact structural-heart papers from behind the institutional firewall. Drop them in `knowledge/papers/inbox/` so the existing indexer processes them into `papers_index.json` for the Claude prompt injection.

**The architecture:**

```
HIGH-IMPACT JOURNAL TOC MONITOR (scheduled, daily)
   ↓ checks new issues of NEJM, JAMA, JACC, Lancet, EHJ, JTCVS, ATS, EJCTS
   ↓
RELEVANCE FILTER
   ↓ Claude scoring: is this paper Valve-Wire-relevant?
   ↓ Filters to ~5-10 papers per week
   ↓
QUEUE TARGET DOIs in data/fetch_queue.json
   ↓
PDF FETCHER (runs locally on your authenticated machine)
   ↓ Opens DOIs via institutional EZproxy URL pattern
   ↓ Uses stored cookies (refreshed weekly via gstack /setup-browser-cookies)
   ↓ Downloads PDFs to knowledge/papers/inbox/
   ↓
EXISTING knowledge/indexer.py PROCESSES THEM
   ↓ Already built — extracts text, metadata, renames, updates papers_index.json
   ↓
NEW PAPERS AUTOMATICALLY INJECTED INTO DAILY DIGEST CLAUDE PROMPTS
   (already wired — papers_index.json is loaded into every prompt)
```

**Files to build:**

| File | Purpose | Time |
|---|---|---|
| `sources/journal_toc.py` | Monitor RSS / DOI feeds for new issues of target journals | ~2 hours |
| `sources/relevance_scorer.py` | Claude prompt: "is this paper Valve-Wire-relevant?" | ~1 hour |
| `data/fetch_queue.json` | List of DOIs to fetch on next run | ~15 min |
| `tools/pdf_fetcher.py` | Open DOIs via EZproxy + download with stored cookies | ~3-4 hours (the hard part) |
| `tools/institution_auth.py` | Cookie management, refresh detection, expiry handling | ~1 hour |
| institutional EZproxy URL pattern documentation | Reverse-engineer or look up your library's proxy pattern | ~30 min research |

**Total: ~1-2 days of focused engineering.**

**The hard part is the EZproxy authentication.** the editor's institution almost certainly uses SSO with MFA. You can't fully automate that. The workaround:

1. Authenticate manually once via your real browser
2. Export the session cookies using the gstack `/setup-browser-cookies` skill (literally built for this — "Import cookies from your real Chromium browser into the headless browse session")
3. The script uses those cookies to fetch PDFs through the proxy until they expire
4. When cookies expire (probably weekly), you re-authenticate manually and re-export

**Where this runs:** NOT GitHub Actions (because cookies would have to be stored as secrets and they expire). Better: a local cron on your Mac, on Eggar (the Mac Mini), or on Beckett (the OpenClaw machine). Or triggered manually from a Telegram command (`/fetch`) when you're sitting at an institution-network machine.

**Risks to manage:**
1. Institutional ToS — most institutional licenses permit personal scholarly use including bulk download. Worth confirming with the institution's library before deploying broadly.
2. Rate limiting from publishers — Elsevier, NEJM Group may flag aggressive download patterns. Mitigate with reasonable delays (1 paper per 30 sec, max ~50/day).
3. Detection from institutional IT — unusual traffic from a fixed external IP authenticating with your credentials may trigger flags. Mitigate by running locally, not from a CI runner.

**CRITICAL:** Don't deploy this BEFORE the AATS launch. If something goes wrong and institutional IT flags your account, you'd lose institutional access right before the launch. **Tier 4 priority — post-launch only.**

- Scheduled fetches (papers, stock data, FDA filings, conference programs)
- Indexing and metadata extraction
- Relevance scoring
- Drafting (first-draft outreach emails, first-draft summaries, first-draft methodology updates)

**Interactive-only** (you must be in the loop):

**The art of running this is keeping the line clear.** Don't let agents make strategic decisions. Don't make yourself sit in the loop for things agents can do.

**The 5 candidate background agents (in priority order):**
